## 1. スケジュール詳細パネルの導入（月間カレンダー・週間タイムライン共通）

### 実装に至った経緯
- カレンダーイベントや週間の記録ブロックをクリックしても、詳細情報の確認導線が弱かった。
- 月間と週間で詳細表示の体験をそろえる必要があった。

### 実装内容
- 選択対象を統一して扱える型を追加。
- イベント・タイマー記録のどちらを選択したかを状態として保持。
- 選択対象が消えた場合に自動クリアする保護ロジックを追加。
- カレンダーイベント、週間タイムラインの各ブロックをクリック可能にして詳細パネルへ接続。

### どこをどう変えたか
- 変更ファイル: src/App.tsx
  - ScheduleDetailSelection 型を追加。
  - selectedScheduleDetail と関連 state を追加。
  - 選択整合性を保つ useEffect を追加。
  - renderScheduleInspector 関連ロジックを追加。
  - 月間/週間のクリックイベントと stopPropagation の制御を追加。
- 変更ファイル: src/App.css
  - カレンダー詳細表示用のレイアウト定義を追加。
  - クリック可能セルの視覚スタイルを追加。

## 14. リスク/注意点（追加整理）

1) スケジュール詳細パネル導入
- リスク/注意点: 選択 state の参照先が消えた時に stale state が残ると描画不整合が起きるため、クリア処理を維持すること。

2) 閉じるボタン縦表示修正
- リスク/注意点: 他のボタン共通クラスに影響する上書きにしないこと（限定セレクタ維持）。

3) 画面サイズ・余白調査
- リスク/注意点: `tauri.conf.json` 変更は初期起動サイズのみ。実ランタイム見え方は CSS の余白設定が支配的。

4) ポップオーバー優先度修正
- リスク/注意点: z-index を上げ過ぎると他モーダルより前面化する恐れ。重なりレイヤー規約を固定すること。

5) 詳細パネルの選択時表示化
- リスク/注意点: 条件分岐ミスで source 不一致時に何も出ない/誤表示の可能性。`calendar` と `weekly` の分岐を保守時に崩さないこと。

6) 月間カレンダークリック導線改善
- リスク/注意点: セル onClick と子要素 onClick のバブリング競合に注意。stopPropagation を維持すること。

7) CSS 構文エラー修正・レスポンシブ補強
- リスク/注意点: CSS は小さな記述漏れで全体崩壊しやすい。大きな変更後は必ずビルド/問題パネルを確認すること。

8) delete_task の FK エラー対応
- リスク/注意点: 子データを物理削除する挙動になるため、監査用途データが必要なら論理削除方針の再検討が必要。

9) 新規プロジェクト色入力修正
- リスク/注意点: ブラウザ/OS で color input 見た目差異があるため、padding/background の上書きは最小限に留めること。

10) 親フォルダ自動追従
- リスク/注意点: ユーザーが意図的にルートを選びたい場面で自動上書きされる可能性がある。必要に応じて追従 ON/OFF を検討。

11) フォルダ情報量の拡張
- リスク/注意点: 既存 DB のマイグレーション漏れがあると起動時エラーにつながる。`PRAGMA table_info` ベースの後方互換処理を維持すること。

12) 確認ダイアログ統一 + 今日ハイライト
- リスク/注意点: confirm コールバックが async の場合、二重実行防止や閉じるタイミングに注意。日付判定はタイムゾーン差分を考慮すること。
